Introduction[edit]

Network Development Kit (NDK) working on DM648 stops functioning in the presence of high network traffic. The device stops receiving packets and does not recover from it.

This article provies changes to the driver source code that helps fix this issue.

Prerequisites[edit]

Familiarity with building a project using Code Composer Studio, connecting to a target DSP using an emulator and running the executable.

Problem Definition[edit]

The NDK provides a TCP/IP stack and associated services that work with DSP/BIOS on DSPs like DM648. The 3 port switch gigabit ethernet subsystem on DM648 provides ethernet packet communication and can be configured as an ethernet switch. A DM648 running as a network host should receive and respond if needed to packets sent to it, from any device (such as a PC) connected to the same network.

A single ISR handles both transmit and receive interrupts from the ethernet subsystem to process the packets. Once an interrupt is issued from the subsystem, the CPU handles it and then acknowledges the interrupt. The subsystem has to wait until it receives an acknowledgment from the CPU before generating the next interrupt. This is done by disabling the interrupts upon entering the ISR and enabling them again at the end of ISR.

But the driver disables only the EMAC interrupts in this process, and other interrupts are generated before a previous interrupt is acknowledged. So, the ISR is re-entered before the previous interrupt is serviced completely, and this messes up the Buffer Descriptors causing errors that stop the NDK to respond to packets.

Hardware and Software Used to Reproduce the Problem[edit]

Hardware[edit]

DM648 Evaluation Module (EVM)

A Windows machine

A Linux machine, preferably with Ubuntu

Software[edit]

On Windows
[edit]

The issue can be reproduced using a sample CCS project from an NDK

Code composer studio: Version 5.4.0.00091 or later (not tested with earlier versions)

BIOS: bios_5_42_00_07 (Windows installer)

NDK: ndk_2_20_06_35

Driver: ndk_2_0_0. For DM648, reference drivers are not included with any NDK releases made after NDK 2.00.

CCS Project: Taken from ndk_2_0_0. Project is located at ndk_2_0_0\packages\ti\ndk\example\network\helloWorld

Note: The project has to be converted to work with CCS 5.4. The project should be updated to link the stack libraries from NDK 2.20.06.35 and driver libraries from NDK 2.0.0.

On Linux
[edit]

Install 'hping3' on Linux machine that can be used to flood the device. If you are using Ubuntu install it with the following command

sudo apt-get install hping3

Procedure to Reproduce the Problem[edit]

The functionality of a DM648 or any DSP running as a network host can be tested using the 'ping' utility from a PC connected to the same network as the DSP. Ping sends one echo request packet every second to the target host and measures the round-trip time. The traffic can be increased by flooding where the target host is overwhelmed with echo packets. Flooding is done using hping3 tool from a Linux machine.

1. Connect the EVM and the linux machine with a network cable

2. Build the sample project, run it from CCS on a Windows machine. The IP address assigned to the EVM will be displayed on the CCS console

3. From the linux machine, perform a ping from a terminal window 

ping <ip_address_of_EVM>

This prints the round-trip time of each packet sent. This lets us know if the board is responding to packets.

4. From another terminal window, use hping3 to flood the EVM with packets at maximum rate 

sudo hping3 --udp --flood <ip_address_of_EVM>

Response to every packet is not displayed on the screen. Now it can be seen from the 'ping' window that the packet round-trip response is delayed. When there is no more response, the ping stops printing the round-trip time, and after sometime something like the following is printed

From <ip_address_of_EVM> Destination Host Unreachable

This means the board has stopped responding to packets. Now, the board does not recover and respond to 'ping' even after terminating the 'hping3' running in the other window.

If this issue is not reproduced with one 'hping3' flooding, increase the traffic by flooding from multiple terminal windows simultaneously or from other computers connected to the network.

Fix[edit]

As mentioned above, the driver disables only the EMAC interrupts resulting in generation of other interrupts when an interrupt is handled. So this problem is fixed by disabling all the CPU interrupts within the ISR. The interrupts are disabled, and after the task within ISR is completed, the interrupts are restored to the state they were in before they were disabled.

To use this fix, please replace the ethdriver.c located in ndk_2_0_0\packages\ti\ndk\src\hal\evmdm648\ethss_dm648 with the file that can be extracted from the zip file below

Media:Ethdriver.zip

The two function calls that are replaced are 

Disable_DSP_Interrupts(Uint32 IntFlag)

Enable_DSP_Interrupts(IntFlag);

These are respectively replaced with 

uint key = disableInterrupts();

restoreInterrupts(key);
