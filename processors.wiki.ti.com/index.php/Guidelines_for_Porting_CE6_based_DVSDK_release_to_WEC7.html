Introduction
[edit]

This page provides general guidelines that one can follow to port the DVSDK package running on CE6 to WEC7. Note that the changes specified here are not exhaustive and depending on your custom changes to DVSDK package, porting effort may vary. The changes listed below were used to port TI provided DVSDK 1.11.00.03 patch 01 release for WEC7 and the binaries generated are shipped as a part of BSP WINCE ARM A8 2.0 release.


Users should first build the required DVSDK package with the right codecserver and tools on a CE6 build machine. Then copy this built package to WEC7 build machine, apply the WEC7 changes as highlighted in this document and re-build the necessary components (LPM, DSPLINK, CMEM, DSHOW). This logic applies for 720p build option as well. To build a DVSDK package in WEC7 with 720p support, they should build the CE6 DVSDK package on a CE6 build machine with TI_ENABLE_720P_SUPPORT enabled. Then copy this built package to WEC7 build machine, apply the WEC7 changes as highlighted in this document and re-build the necessary components (LPM, DSPLINK, CMEM, DSHOW).

DVSDK 1.11 Download Link: http://software-dl.ti.com/sdoemb/sdoemb_public_sw/wince_sdk/latest_1_x/index_FDS.html


Limitations[edit]

1. Ony DSHOW video decoders (H264, MPEG4, MPEG2) have been ported for WEC7 BSP WINCE ARM A8 02.00.00 release. Porting encoders is an exercise left for the reader

2. DMAI apps have not been ported.

3. C6Accel has not been ported.

4. Only the ARM side components have been re-built where necessary. Components that just produce libs (and not exe or dlls) have been left untouched and ARMV4I libs have been used.

5. VRFB needs to be disabled if you want to play video file based on DSP codec using Silverlight Video Player. The reasoning behind this is as follows: Silverlight based Video Player needs compositor to be enabled. When compositor is enabled, then the system requests a number of distinct hardware surfaces even for small icons. When VRFB is also enabled, the number of hardware surfaces available at a time is limited to 12 (VRFB spec). So with compositor and VRFB enabled, the hardware surfaces are soon depleted and there are none left for the video surfaces. Now, the DSP based codec are optimized assuming that the video buffers would be hardware buffers and not system buffers, hence VRFB needs to be disabled.

     Workaround: If VRFB feature is desired, use Microsoft provided Dshow player (ceplayit) which is not silverlight based. Disable Silverlight and compositor in catalog and enable VRFB.

6. Video playback fails on LCD and display goes blank (SDOCM00081313). This is due to the fact:
            a. Display on LCD is in portrait mode by default.
            b. VRFB is disabled in the default OS-design due to above limitation#2
            c. Silverlight based video player plays video file only in landscape mode and hence needs rotation.
            d. DMA based rotation is not sufficient enough for this purpose.
      Workaround: Use Microsoft provided Dshow player (ceplayit) which is not silverlight based for LCD based demos. See C:\WINCE700\public\directx\sdk\samples\dshow\Players\ceplayit\readme.txt for more details on this player.

Prelink.bat
[edit]

LPM

CE6 version of DVSDK package uses prebuilt version of LPM. Since LPM is an ARM driver, we need to rebuild it for WEC7 release. To enable prelink.bat to build LPM driver, add the following changes:

:lpm

@set LPM_PROJDIR=%LPM_INSTALL_DIR%\packages\ti\bios\power\drivers\wince\%MY_PLATFORM%\lpm\
@set LPM_DRV=%LPM_PROJDIR%\obj\%_TGTCPU%\%WINCEDEBUG%\lpmdrv
@set LPM_REG=%LPM_PROJDIR%\lpmdrv.reg


'''@if [%BUILD_LPM%]==[1] (
@echo Building LPM...
@pushd %LPM_PROJDIR%
@build %DVSDK_BUILDOPT% -j %DVSDK_LOGDIR%\lpm
@popd
) else (
@echo Bypassing LPM build...
)
'''

@echo == Copying LPM driver...
%COPYCMD% %LPM_DRV%.dll %DVSDK_OBJDIR%
%COPYCMD% %LPM_DRV%.map %DVSDK_OBJDIR%
%COPYCMD% %LPM_DRV%.pdb %DVSDK_OBJDIR%

@REM copy bib entries
@echo MODULES >> %DVSDK_INSTALL_DIR%\DVSDK.bib
@echo IF SYSGEN_DSPLINK >> %DVSDK_INSTALL_DIR%\DVSDK.bib
@echo lpmdrv.dll $(_FLATRELEASEDIR)\lpmdrv.dll NK SHK >> %DVSDK_INSTALL_DIR%\DVSDK.bib
@echo ENDIF SYSGEN_DSPLINK >> %DVSDK_INSTALL_DIR%\DVSDK.bib

@type %LPM_REG% >> %DVSDK_INSTALL_DIR%\DVSDK.reg

CODECSERVER

CodecServer runs on DSP and hence doesnt need any porting effort. If you are not planning to upgrade the XDC toolchain that supports WEC7 release, I would highly recommend that you do not rebuild codecserver here and use the codecserver as is from CE6 release. Please comment out the lines that set the BUILD_CODECSERVER variable in prelink.bat. Similar line will have to be commented out in setbuild.bat

:codecserver
@set CS_X64P=%CODEC_INSTALL_DIR%\packages\ti\sdo\server\cs\bin\cs.x64P
@set CS_BIB=%DVSDK_INSTALL_DIR%\src\misc\CodecServer.bib
@set CS_LOGFILE=%DVSDK_INSTALL_DIR%\logs\codecserver.log

@rem **** Force Codec Server build if DSP Link was built ****
@rem **** even if /nocs was specified ****
'''REM @if [%BUILD_DSPLINK%]==[1] (
REM set BUILD_CODECSERVER=1
REM )'''
@if [%BUILD_CODECSERVER%]==[1] (

@type %CS_BIB% >> %DVSDK_INSTALL_DIR%\DVSDK.bib

DSHOW

The Timm layer (part of TI decode/encode filters) is built in 2 parts - first xdc toolchain gathers relevant information from CodecServer and generates intermediate C files. Then using WinCE 6.0 or WEC 7 toolchain, we build the timm driver along with the intermediate C files. Since we are not changing and rebuilding CodecServer, we dont need to do the first step here. We can use the intermediate files as is from CE6 release.

Note: If you plan to change the codecserver for WEC7 release, then either you will need to get the xdc toolchain that supports WEC7 (steps for which are out-of-scope for this document) or build the new codecserver and dshow in CE6 and then use this document to port it for WEC7.
