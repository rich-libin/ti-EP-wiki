Introduction[edit]

Note: : The following instructions are for users using add-on version older than 4.0.3.0.1.0

The following guide contains detailed information on how to port Bluetopia Platform Manager, the Bluetooth Protocol Stack provided by Stonestreet One, to a platform other than AM335x EVM or BeagleBone. Note that if you already using an AM335x EVM or BeagleBone White then there is no need to port the stack and you can download a ready-to-use distribution from:

  • WiLink 8 Linux Add-On Packages for Sitara AM335X

The packages found in the link above above were cross-compiled with the Linaro GCC Toolchain that is included in Texas Instrument's AM335x SDK v08.00 in Ubuntu 12.04. If you are using a different cross-compiler or a different version of Texas Instrument's SDK then you may need to contact Texas Instrument's to determine if the packages above are binary-compatible with your development environment.

General Information[edit]

If you haven't already done so, download a distribution from the link above and install it using the instructions found in the Bluetopia Platform Manager User's Guide. In this guide we will port the Bluetopia Platform Manager distribution to your specific platform so that you can get started using Bluetooth with your WL18xx device.

The distribution you have just downloaded and installed contains libraries in /opt/ss1/BluetopiaPM/Bluetopia/lib that Bluetopia Platform Manager links to when it is re-built. In /opt/ss1/BluetopiaPM/Bluetopia/lib there are 2 library files that are relevant to this guide: libBTPSVEND.a and libSS1BTPS.a. The BTPSVEND library contains platform-specific code that will need to be modified to suit your platform. The SS1BTPS (Stonestreet One Bluetooth Protocol Stack) library contains the platform-independent Bluetooth stack. Both of these files are static libraries, therefore if the libraries are changed, any applications that are linked against them must be re-compiled.

The following guide assumes that Bluetopia Platform Manager has been installed to its default location on the local machine (/opt/ss1). If it is not, you will have to replace /opt/ss1 with the directory where Bluetopia Platform Manager has been installed on your local machine.

BTPSVEND (Bluetooth Protocol Stack Vendor-Specific Code)[edit]

The source code for the /opt/ss1/BluetopiaPM/Bluetopia/lib/libBTPSVEND.a library is located in the /opt/ss1/BluetopiaPM/Bluetopia/sample/SS1BTVEN directory, as well as a makefile, SS1BTVEN.mak, that you can use to re-build the library. The source code file BTPSVEND.c contains functions that the SS1BTPS library will call. The API for the BTPSVEND library can be found in /opt/ss1/BluetopiaPM/Bluetopia/include/BVENDAPI.h. BVENDAPI.h contains the following functions:

HCI_VS_InitializeBeforeHCIOpen(): This function is called from the SS1BTPS library before the Serial Port/UART Driver is opened. It can be used to specify the Serial Port/UART Baud Rate, COM Port Number, or to enable the Bluetooth Controller.
HCI_VS_InitializeAfterHCIOpen(): This function is called just after the Serial Port/UART Driver has been opened, but before the SS1BTPS library has begun Serial Port communication with the Bluetooth Controller. Typically nothing is done in this function.
HCI_VS_InitializeBeforeHCIReset(): This function is called just before Serial Port/UART communication with the Bluetooth Controller begins. After this function exits the SS1BTPS library will send an HCI Reset Command to the Bluetooth Controller to initialize the Bluetooth Controller. Typically nothing is done in this function.
HCI_VS_InitializeAfterHCIReset(): This function is called just after the HCI Reset Command has been issued, and the Bluetooth Controller has been successfully initialized. Successful in this case means the Bluetooth Controller received and acknowledged the HCI Reset Command. It is in this function that the Bluetooth Controller's BTS Initialization Script can be sent to the Bluetooth Controller. The purpose of the BTS Initialization Script is discussed in more detail below.
HCI_VS_InitializeBeforeHCIClose(): This function is called just before the stack is shutdown. Typically nothing is done in this function.
HCI_VS_InitializeAfterHCIClose(): This function is called after the stack has been shutdown. This function can be used disable the Bluetooth Controller if desired, see more on disabling the Bluetooth Controller below.

For more information regarding the specific parameters passed to these functions see the comments in /opt/ss1/BluetopiaPM/Bluetopia/include/BVENDAPI.h.

The Bluetooth Controller's Initialization Script[edit]

The directory /lib/firmware/ contains one or more BTS scripts that are loaded on to the Bluetooth Controller as part of the Bluetooth initialization process. The /lib/firmware/ directory may contain multiple BTS scripts, only one script is loaded on to the controller, and the BTS Script that is appropriate for the current Bluetooth Controller is selected by first querying the Device Version Information from the Bluetooth Controller.

The BTS Scripts are formatted in a TI-proprietary format, and contains several HCI Commands. The appropriate BTS Script is executed and the HCI commands are sent to the Bluetooth Controller after the Bluetooth Controller has successfully acknowledged an HCI Reset Command. The HCI Commands that parsed from the script and then sent to the Bluetooth Controller can and typically do many things, such as:

  1. Configure the Bluetooth Controller's Baud Rate
  2. Apply patches to the Bluetooth Controller's Software
  3. Configure the Bluetooth Controller to the Desired State

Item 1, Configure the Bluetooth Controller's Baud Rate: When a Bluetooth Controller's enable pin is pulled low the Bluetooth Controller is disabled. When the enable pin is high the Bluetooth Controller is enabled. When the Bluetooth Controller is first enabled it is expecting data at a baud rate of 115,200. Therefore, every time the Bluetooth Controller is re-enabled it must be initialized at 115,200. A baud rate of 115,200 is not fast enough for many applications, especially for audio streaming, therefore it is possible to send an HCI Command to the Bluetooth Controller to change its baud rate after the HCI Reset Command has been successfully issued. Sometimes this Change-Baud Rate command is embedded within the BTS Initializations Script, and sometimes it is done manually after the Bluetooth Controller has been initialized in HCI_VS_InitializeAfterHCIReset().

Item 2, Apply Patches to the Bluetooth Controller's Software: The Bluetooth Controller may have software fixes and/or software updates that need to be applied after it is initialized. It is possible for the initialization script to contain HCI Commands to update its software. In this respect the initialization script can be thought of as a Software Service Pack.

Item 3, Configure the Bluetooth Controller to the Desired State: The Bluetooth Controller may need to be configured to meet the needs of the target application. For example, if the Bluetooth Controller needed to be used with Bluetooth Low Energy support, the patch might contain an HCI Command to enable Low Energy. The BTS Initialization Script contains the necessary HCI Commands to configure the Bluetooth Controller to the desired state.

Bluetooth Pins used by the Stack[edit]

Bluetooth Enable: This pin controls if the Bluetooth Controller is in reset state or if is enabled. When this pin is low the Bluetooth Controller is disabled, when the pin is high the Bluetooth Controller is enabled. This pin is typically referred to in schematics as BTEN, BTRST, NSHUTDN, or BTSHUT. The stack controls this pin with the code in the BTPSVEND library.
UART RTS (Request-to-Send): This pin is an output of the stack's processor and is used to tell the Bluetooth Controller one of two things: 1) that the stack has data it would like to send to the Bluetooth Controller, or 2) that the stack is able to receive more data. This pin is held low when the stack and Bluetooth Controller are both idle, i.e. when neither the stack or Bluetooth Controller is sending or receiving data. This pin is high when the UART driver's buffer is full and also when the UART driver is closed. The stack does not directly control this pin, it is controlled in the kernel by the UART driver.
UART CTS (Clear-to-Send): This pin is an output of the Bluetooth Controller's processor and is used to tell the stack one of two things: 1) that the Bluetooth Controller has data that it would like to send the stack, or 2) that the Bluetooth Controller is free to receive data. This pin is held low when the stack and Bluetooth Controller are both idle. The Bluetooth Controller sets this pin high when it is busy and is not able to receive data. The stack does not control this pin, it is controlled by the Bluetooth Controller.
UART Tx (Transmit): The stack uses this pin to send data to the Bluetooth Controller. The stack will not send data when the UART CTS pin is high. The stack does not directly control this pin, it is controlled in the kernel by the UART driver.
UART Rx (Receive): The Bluetooth Controller uses this pin to send data to the stack. The Bluetooth Controller will not send data to the stack when the UART RTS pin is high. The stack does not control this pin, it is controlled by the Bluetooth Controller.

Modifying BTPSVEND for a New Platform[edit]

The version of BTPSVEND.c that you have contains several measures which should simplify porting BTPSVEND to a new Platform.

The only parameters in BTPSVEND.c that will likely need to change are as follows:

BLUETOOTH_HCI_DEV_TTY_COM_PORT_NUMBER: This definition specifies the which COM port number should be used. If this value is set to 1 the Bluetooth Stack will use the COM port /dev/ttyO1, and if this value is set to 2 Bluetooth Stack will use the COM port /dev/ttyO2.
BLUETOOTH_HCI_TARGET_BAUD_RATE: This definition specifies what HCI Baud Rate to use when communicating with the Bluetooth Controller. Note that for audio applications the minimum baud rate that can be used is 921,600.
Enable_Bluetooth_Controller(): This function is called when the Bluetooth Controller should be enabled, i.e. when the Bluetooth Enable pin should be set high. This function will likely need to be modified to set the appropriate GPIO pin high for your platform.
Disable_Bluetooth_Controller(): This function is called when the Bluetooth Controller should be disabled, i.e. when the Bluetooth Enable pin should be set low. This function will likely need to be modified to set the appropriate GPIO pin low for your platform.

Rebuilding BTPSVEND for a New Platform[edit]

The distribution you have downloaded contains executables in the /opt/ss1/BluetopiaPM/Bluetopia/bin/ directory. All of these libraries were compiled and then linked against the libBTPSVEND.a and libSS1BTPS.a libraries stored in the /opt/ss1/BluetopiaPM/Bluetopia/lib/ directory. To modify BTPSVEND to support your platform you will need to re-build the BTPSVEND library. And because BTPSVEND is a static library, you will also need to re-build any libraries that were linked against it, this includes the samples in /opt/ss1/BluetopiaPM/Bluetopia/bin, and the PM Server binary, SS1BTPM, in /opt/ss1/BluetopiaPM/bin/SS1BTPM (more details about building SS1BTPM is given below).

As mentioned earlier, the source code for the /opt/ss1/BluetopiaPM/Bluetopia/lib/libBTPSVEND.a can be found in the /opt/ss1/BluetopiaPM/Bluetopia/sample/SS1BTVEN directory. Inside of that directory there is a Makefile named BTPSVEND.mak. After making changes to BTPSVEND.c, libBTPSVEND.a can be re-built by executing the following: 

$ make -f BTPSVEND.mak

This will create a new libBTPSVEND.a in the current directory.

Note: This Makefile, and all Stonestreet One-provided Makefiles, use environment variables named GLOBCFLAGS, GLOBINCLDDIRS, GLOBLDFLAGS, and GLOBLDLIBS to configure the build environment. These environment variables must set appropriately when cross-compiling BTPSVEND or any samples applications.

Now that libBTPSVEND.a has been re-built, next you can rebuild a Bluetopia sample to test your BTPSVEND changes. The Makefiles (*.mak) included with the samples all link against the libBTPSVEND.a library stored in the /opt/ss1/BluetopiaPM/Bluetopia/lib/ directory. So first copy the newly compiled libBTPSVEND.a library to the /opt/ss1/BluetopiaPM/Bluetopia/lib/ directory (note that you may want to back up your previous library in case you ever need to revert to the original platform). If your current working directory is /opt/ss1/BluetopiaPM/Bluetopia/sample/SS1BTVEN, you can copy the library to /opt/ss1/BluetopiaPM/Bluetopia/lib/ using the following: 

$ cp ./libBTPSVEND.a ../../lib/

Now that you have copied libBTPSVEND.a to the /opt/ss1/BluetopiaPM/Bluetopia/lib/ directory you can now continue with re-building the sample. As an example we'll re-build LinuxSPP, however, note that any of the samples can be re-built using a similar procedure to the one explained here.

Next, navigate to the LinuxSPP sample directory: 

$ cd ../LinuxSPP/

Now re-compile the sample using: 

$ make -f LinuxSPP.mak

You will now have a new LinuxSPP executable that is linked against your new libBTPSVEND.a library. You can test the sample by copying it to your target device and then executing it on the target using: 

# ./LinuxSPP 1 0 <comm_port_number> <baud_rate>

If all went well the LinuxSPP demo will boot to an "SPP>" prompt. If it does, congratulations! You have successfully ported the BTPSVEND library to your platform. If it does not, double-check the code in BTPSVEND.c for any mistakes. If none are found, please contact Stonestreet One or Texas Instruments for additional support.

Now that you have successfully initialized a Bluetopia application, the next step is to re-build the PM Server Executable. Bluetopia Platform Manager's make files use the environment variables BLUETOPIA_PATH and BLUETOPIA_ROOT_PATH to reference Bluetopia's Libraries and Bluetopia's Header Files. These environment variables must therefore be defined before attempting to re-build the PM Server. If you have installed Bluetopia Platform Manager to the default directory on your local machine, /opt/ss1, then set the environment variables using the following: 

$ export BLUETOPIA_PATH=/opt/ss1/BluetopiaPM/Bluetopia
$ export BLUETOPIA_ROOT_PATH=/opt/ss1/BluetopiaPM/Bluetopia

Next, re-build the PM Server (SS1BTPM) using: 

$ cd /opt/ss1/BluetopiaPM/server/
$ make -f SS1BTPM.mak clean; make -f SS1BTPM.mak

A new PM Server executable will be created in the current directory. You can optionally copy SS1BTPM to /opt/ss1/BluetopiaPM/bin to replace the copy of SS1BTPM that was built for the original platform, or you can simply execute the SS1BTPM binary stored in the current directory when running Bluetopia Platform Manager on the target device. Note that there is no need to re-build any executables in the /opt/ss1/BluetopiaPM/bin directory other than the PM Server (SS1BTPM). The remaining executables are Bluetopia Platform Manager demos, and, unlike Bluetopia demos in /opt/ss1/BluetopiaPM/Bluetopia/bin, do not link with libBTPSVEND.a and therefore do not need to be re-compiled.

The PM Server[edit]

Module Configuration[edit]

The files included in the Platform Manager archive by default include a large set of modules and Bluetooth profiles, it is likely however that only a subset of the modules/profiles is needed for a specific project. Code for each module is compiled into the PM Server Executable (SS1BTPM) and each module is initialized in the executable. The source code file /opt/ss1/BluetopiaPM/btpmmodc/server/BTPMMODC.c contains a table, ModuleHandlerList, of function pointers that can be modified to prevent a module from being initialized in the PM Server. To remove a module simply delete the line containing references to that module in the ModuleHandlerList table, and re-compile SS1BTPM. Use the following to recompile SS1BTPM: 

$ vim /opt/ss1/BluetopiaPM/btpmmodc/server/BTPMMODC.c
(remove references to unused modules)
$ cd /opt/ss1/BluetopiaPM/btpmmodc/server/
$ make -f BTPMMODC.mak clean; make -f BTPMMODC.mak
$ cd ../../ss1btpm/server/
$ make -f SS1BTPM.mak clean; make -f SS1BTPM.mak

You can optionally remove:

  1. The unused data structures in BTPMMODC.c
  2. The unused includes in BTPMMODC.c
  3. The unused include paths in BTPMMODC.mak
  4. The unused includes and libraries from SS1BTPM.mak


Pairing Mode[edit]

SS1BTPM is compiled to support secure simple pairing as the default configuration. If legacy pairing is needed instead, for example so that the user is forced to enter a fixed pin code, then it is possible to configure and recompile SS1BTPM to only support legacy pairing. To do this open /opt/ss1/BluetopiaPM/ss1btpm/server/BTPMSRVR.c in a text editor. Near the bottom of the file modify the assignment: 

DefaultInitializationData.DefaultPairabilityModeSSP = TRUE;

and change TRUE to FALSE. 

DefaultInitializationData.DefaultPairabilityModeSSP = FALSE;

After making this change next rebuild SS1BTPM using the following: 

$ make -f SS1BTPM.mak clean; make -f SS1BTPM.mak

SS1BTPM is now compiled to support only legacy pairing. Next copy SS1BTPM to your target and run the executable. You will notice that the next time a device pairs with the target that the demo application will prompt the user for a pin code response. When this occurs respond with the PINCodeResponse command, for example "PINCodeResponse 0000". The pin code used, "0000" in this example, must be the same on both devices.

Note: Keep in mind that Legacy Pairing is generally not as secure as Secure Simple Pairing. If security is a concern for your application then legacy pairing is not recommended. For more details on the security of Bluetooth pairing modes refer to the core specification which can be found on the Bluetooth SIG's website. Also note that, in the default configuration where secure simple pairing is enabled, legacy pairing is still supported. If a remote device attempts to connect to the target and the remote device only supports legacy pairing, the target will fall back to legacy pairing.


PM Clients[edit]

The source file /opt/ss1/BluetopiaPM/btpmmodc/server/BTPMMODC.c specifies which modules are initialized when a client application initializes (when BTPM_Initialize() is called). To remove a module simply delete the line containing references to that module in the ModuleHandlerList table, and re-compile the SS1BTPM Client library: 

$ vim /opt/ss1/BluetopiaPM/btpmmodc/client/BTPMMODC.c
(remove references to unused modules)
$ make -f BTPMMODC.mak clean; make -f BTPMMODC.mak
$ cd ../../ss1btpm/client/
$ make -f SS1BTPM.mak clean; make -f SS1BTPM.mak

You can optionally remove:

  1. The unused includes in BTPMMODC.c
  2. The unused include paths in BTPMMODC.mak
  3. The unused includes and libraries from SS1BTPM.mak
