Understanding the CCS Loader and dual-core TI-RTOS applications

From Texas Instruments Wiki
Jump to: navigation, search

Dual-core TI-RTOS applications[edit]

F28M35x (Concerto) and F28M36x (Aumento) are two devices that TI-RTOS supports right out of the box. Both devices have a C28 core alongside an M3. An example TI-RTOS project that uses both cores is the F28M35H52C1/F28M36P63C2 Demo project that ships with TI-RTOS. These projects use IPC to facilitate communication between both cores.

CCS Loader with dual-core projects[edit]

The procedure for running these dual-core applications on CCS includes loading the M3 application followed by the the c28 application and then running the M3 before the c28. Depending on CCS settings this procedure could lead to timing issues in your application especially if it uses the BIOS_getCpuFreq to configure some peripheral (I2C, SPI, etc). The reason is as follows:

SYS/BIOS sets the cpu frequency for the M3 as part of its initialization, which occurs before main(). When your CCS "Auto Run and Launch Options" is set to run to Main after a program load and you load M3 first, the CPU is set to the "BIOS.cpuFreq" settings (from the .cfg file). When the c28 is loaded next, the CCS loader re-configures the cpu clock divider since the c28 runs at 1/2 the frequency of the M3. The result of this is, when you run your application, the M3 cpu frequency as specified by SYS/BIOS config gets divided by 2 and therefore the BIOS_getCpuFreq function will not return the actual cpu frequency. The effects of this can be seen in peripherals for example the UART configured to use the BIOS_getCpuFreq value not working correctly since its configured with a frequency different from the actual cpu frequency.

Work arounds[edit]

Note that this issue is only seen when loading your application using CCS. When booting from Flash this issue will not occur. To boot dual-core applications from flash, you will have to edit the M3 applications configuration script using XGCONF and check the "Initiate boot of the C28 processor" box as shown below.

When loading with CCS there are three ways you can work-around this possible problem.

First you can follow this load/run sequence:

  1. Load C28 application
  2. Load and run M3 application
  3. Perform a CPU reset and restart C28

A second solution, is to disable the option of loading and running to Main in the CCS Auto Run and Launch Option for the M3. The image below show how to achieve this.

The third solution, is to tell the C28 loader to set the SYSDIVSEL register to match your PLL configuration. By default, when configuring the system PLL to run at 150 MHz you need to set SYSDIVSEL to 0. See below.
